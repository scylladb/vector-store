# Releasing binaries from vector-store repository

## TL;DR

```bash
./scripts/release X.Y.Z
```

This single command validates dependencies, sets up Docker QEMU (for image
packaging), creates and pushes the annotated tag, builds all artifacts
(binaries, docker images, SBOMs), and uploads everything to GitHub and Docker
Hub. When run from a fork, Docker Hub pushes are skipped automatically — only
GitHub release assets are uploaded.

## Releasing vector-store

Release binaries are cross-compiled inside a Docker container using the
appropriate GCC cross-toolchain (e.g. `gcc-aarch64-linux-gnu` for arm64).
QEMU is only needed for Docker image packaging (`docker build --platform`),
not for binary compilation. You can use `scripts/prepare-docker-qemu` to
register QEMU handlers in the Docker engine.

Release version is based on git annotated tags. You can create a new tag using:

```bash
git tag -a X.Y.Z -m "Release version X.Y.Z"
git push origin X.Y.Z
```

GitHub doesn't support creating annotated tags, so you must do them from local
environment. You can add additional metadata to the tag, for example
`X.Y.Z-dev` for development releases.

Script `scripts/run-with-release-toolchain` is a Docker wrapper that derives the
version from annotated tags using `git describe --dirty` and patches `Cargo.toml`
accordingly. It handles cross-compilation setup (installing cross-toolchains,
setting `CARGO_BUILD_TARGET`, linker, and C/C++ compiler environment variables).
You do not need to run it manually — it is called automatically by
`scripts/build-release`.

To build the release you need to git clone repository and switch to the desired
version tag. Then from the root of the repository run:

```bash
./scripts/build-release
```

Optionally, you can pass a version explicitly to override automatic detection
from git tags:

```bash
./scripts/build-release 1.6.0
```

This will create a tar.gz release in `target/<arch>/release/`,
per-architecture docker images, and SBOM files in the `SBOM/` directory.

The build script generates SBOM files in CycloneDX JSON format:
- `SBOM/vector-store-{version}.cdx.json` — Cargo dependencies SBOM generated
  by `cargo-cyclonedx`
- `SBOM/vector-store-docker-{version}-{arch}.cdx.json` — per-architecture
  Docker image SBOM generated by `syft`, covering OS-level packages from the
  base image (one per architecture: amd64, arm64)

After building the release, use the upload script to publish artifacts. If the
GitHub release does not exist yet, it will be created with auto-generated
release notes. If it already exists, assets are uploaded to the existing release:

```bash
./scripts/upload-release 1.6.0
```

This will upload all tar.gz archives and SBOM files to the GitHub release
matching the current version tag, push per-architecture docker images, create
and push the multi-platform docker manifest. It requires the `gh` CLI to be
installed and authenticated, and Docker to be logged in to Docker Hub
(`docker login`).

> **Note:** Docker image pushes only happen when the origin remote points to
> `scylladb/vector-store`. When running from a fork, Docker pushes are skipped
> and local images remain available for verification.

### Prerequisites

- Docker with `buildx` support
- QEMU handlers registered for multi-platform Docker images (see
  `scripts/prepare-docker-qemu`)
- `cargo-cyclonedx` installed (`cargo install cargo-cyclonedx`)
- `syft` installed (see https://github.com/anchore/syft)
- `gh` CLI installed and authenticated (see https://cli.github.com/)
